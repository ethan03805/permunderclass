# Package C3: Auth Session Redirect Fix

## What Changed

Fixed the sign-in redirect bug where a user could successfully authenticate, land on `/`, and still see the signed-out homepage until a second navigation. The root cause was Turbo handling session-changing auth form submissions against cached anonymous pages.

The sign-in form, enrollment confirmation form, and sign-out button now opt out of Turbo so those transitions use a full browser request cycle and render the correct signed-in or signed-out state immediately after the redirect.

## Files Added or Modified

- `app/views/sessions/new.html.erb` - disabled Turbo on the sign-in form
- `app/views/enrollments/show.html.erb` - disabled Turbo on the enrollment confirmation form
- `app/views/shared/_site_nav.html.erb` - disabled Turbo on the sign-out form generated by `button_to`
- `test/integration/sign_in_flow_test.rb` - added coverage for the non-Turbo sign-in form and the signed-in nav after redirect
- `test/integration/enrollment_flow_test.rb` - added coverage for the non-Turbo enrollment form and the signed-in nav after redirect
- `docs/package-c3-auth-session-redirect-fix.md` - this task note

## How The Change Was Verified

- Reproduced the bug in a real browser flow against `http://127.0.0.1:3000`:
  - visited `/`
  - opened `/sign-in`
  - signed in with a valid TOTP code
  - confirmed the pre-fix behavior could land on `/` with the signed-out nav still visible
- Re-ran the same browser flow after the fix and confirmed:
  - sign-in lands on `/` with the signed-in nav immediately visible
  - sign-out lands on `/` with the signed-out nav immediately visible
- Ran targeted auth tests:
  - `docker compose run --rm app bin/rails test test/integration/sign_in_flow_test.rb test/integration/enrollment_flow_test.rb`
- Ran lint:
  - `docker compose run --rm app bin/lint`
